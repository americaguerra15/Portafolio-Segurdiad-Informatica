Lab 2

SQL Injection vulnerability allowing login bypass

Información del laboratorio

Nivel: Apprentice

Tipo de SQLi: Error-based / Classic SQL Injection

Objetivo

Explotar una vulnerabilidad SQL Injection en el formulario de login para autenticarse como el usuario administrator sin conocer su contraseña real, modificando la lógica de la consulta SQL de autenticación y logrando acceso no autorizado a la aplicación.

Vulnerabilidad

El formulario de login construye una consulta SQL dinámica concatenando directamente los valores de los campos username y password sin sanitización ni uso de sentencias preparadas.

La consulta típica es:

SELECT * FROM users 
WHERE username = 'input_username'
AND password = 'input_password'

Al inyectar un cierre de comilla simple (') seguido de un comentario SQL (--) se altera la lógica de la consulta.

El comentario SQL elimina la verificación de contraseña, permitiendo autenticarse como administrador si el usuario existe en la base de datos.

Procedimiento

Paso 1: Ir a la sección My Account.

Paso 2: En el campo Username ingresar:

administrator'--

El cierre de comilla simple termina la cadena del username y -- comenta el resto de la consulta SQL.

Paso 3: En el campo Password ingresar cualquier valor:

123

Paso 4: Enviar el formulario presionando Log in.

La consulta SQL resultante se convierte en:

SELECT * FROM users 
WHERE username = 'administrator'--' 
AND password = '123'

El comentario SQL -- elimina la verificación de contraseña, por lo que la consulta se reduce a: 

SELECT * FROM users WHERE username = 'administrator'

Si el usuario existe, el sistema concede acceso como administrador, logrando un bypass de autenticación.

Explicación del payload

El payload utilizado fue:

Username: administrator'--
Password: cualquier valor
  • El carácter ' cierra la cadena literal del campo username.
  • El operador -- inicia un comentario SQL.
  • Todo el código posterior de la consulta es ignorado.
  • La consulta solo verifica si existe el usuario administrator.

Esto permite autenticarse sin conocer la contraseña real, aprovechando la concatenación insegura de entradas en la consulta SQL.

Mitigación recomendada

  • Implementar consultas parametrizadas o prepared statements.
  • Validar estrictamente las entradas de usuario (regex, longitud máxima).
  • Evitar caracteres peligrosos como ' o --.
  • Almacenar contraseñas usando hashing seguro como: bcrypt, Argon2 o PBKDF2.
  • Aplicar el principio de mínimo privilegio en la base de datos.
  • Implementar un Web Application Firewall (WAF) para detectar patrones de SQL Injection.
