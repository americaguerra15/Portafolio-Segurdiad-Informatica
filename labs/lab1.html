Lab 1

SQL injection vulnerability in WHERE clause allowing retrieval of hidden data

Información del laboratorio

Nivel: Apprentice

Tipo de SQLi:

  • SQL Injection basada en manipulación lógica (Boolean-Based SQL Injection)
  • In-band SQL Injection
  • Boolean-based SQLi
  • SQLi en parámetro GET
  • SQLi en cláusula WHERE

La vulnerabilidad consiste en modificar la lógica de la consulta SQL agregando una condición siempre verdadera (OR 1=1), lo que provoca que la cláusula WHERE devuelva todos los registros de la tabla.

Objetivo

El objetivo fue modificar la cláusula WHERE para que la condición evaluara como verdadera en todos los casos, permitiendo recuperar datos que normalmente no son visibles para el usuario.

Procedimiento

Primero ingresamos al laboratorio desde Web Security Academy y copiamos la URL generada. El objetivo era explotar una vulnerabilidad de SQL Injection en la cláusula WHERE para recuperar datos ocultos.

Al cargar la página observamos una tienda con filtros por categoría como:

  • Clothing
  • Corporate gifts
  • Food & Drink
  • Lifestyle

Posteriormente abrimos Burp Suite Community Edition y nos dirigimos a:

  • Proxy → Intercept
  • Activamos Intercept is on
  • Seleccionamos Open browser

Esto abre el navegador interno de Burp, permitiendo interceptar el tráfico HTTP.

En el navegador interno pegamos la URL del laboratorio y Burp interceptó automáticamente la petición. Luego seleccionamos Forward all para permitir que la solicitud llegara al servidor.

Después seleccionamos la categoría: Corporate gifts

Burp interceptó la siguiente solicitud: 

GET /filter?category=Corporate+gifts HTTP/2

Este parámetro es utilizado para filtrar productos y probablemente se inserta en una consulta SQL como: 

SELECT * FROM products WHERE category = 'Gifts' AND released = 1

Si el input no se valida correctamente, es posible realizar SQL Injection.

Para modificar manualmente la solicitud:

  • Clic derecho en la petición interceptada
  • Seleccionar Send to Repeater
  • Abrir la pestaña Repeater

En Repeater modificamos la URL agregando el payload: 

'+OR+1=1--

Quedando la petición final así: 

GET /filter?category=Corporate+gifts'+OR+1=1-- HTTP/2

Después de enviar la solicitud modificada:

  • La respuesta HTTP fue 200 OK
  • Se mostraron productos ocultos
  • El laboratorio marcó el estado como Solved

Esto confirma que la aplicación era vulnerable y que la inyección fue exitosa.

Mitigación recomendada

Para prevenir este tipo de vulnerabilidades es importante evitar que los datos ingresados por el usuario se interpreten como parte del código SQL.

Una de las medidas principales es utilizar consultas parametrizadas (prepared statements), ya que separan el código SQL de los datos ingresados. De esta forma, aunque un atacante introduzca código malicioso, este será tratado únicamente como datos.

También es recomendable implementar:

  • Validación y sanitización de entradas
  • Restricción de valores permitidos en parámetros
  • Aplicar el principio de mínimos privilegios en la base de datos
